ZenWave SDK is a set of tools to convert your <strong className="orange">Domain Models</strong> into working software and tests.

It is designed to be modular, configurable and extensible. It uses the standard java classpath as loading mechanism, so you can extend its functionality adding your own custom plugins. Visit [ZenWave SDK](/zenwave-sdk) for all configuration options.

Based on <strong className="orange">Domain Driven Design (DDD)</strong> and <strong className="orange">API-First</strong> principles for <strong className="orange">Event Driven Microservices</strong>.

<Image alt="Speed Feedback Loop" dark={speedFeedbackLoopDark} light={speedFeedbackLoopLight} style={{marginTop: '15px'}}/>

ZenWave SDK can generate code from a mix of different models including:

- [ZDL Domain Language](/docs/event-driven-design/zenwave-domain-language) as <strong className="orange">Ubiquitous Language</strong>. You can describe the core of your Bounded Context, as well as how it connects to external systems through different adapters and APIs.
- [AsyncAPI](https://www.asyncapi.com/docs/getting-started/coming-from-openapi): Industry de-facto standard to describe Event-Driven Architectures for Message-based APIs.
- [OpenAPI](https://swagger.io/specification/): Industry standard for Request-Response Architectures with REST APIs.

Using ZenWave Domain Language as <strong className="orange">Ubiquitous Language</strong> for modeling and describing Bounded Contexts: aggregates, entities with their relationships, services, value objects, commands and events.

ZenWave Domain Language started as an *extended subset* of [JHipster Domain Language (JDL)](https://www.jhipster.tech/jdl/intro) that let you describe your entities and relationships.

Using ZDL Domain Language as **Ubiquitous Language** for <strong className="orange">Data on the Inside</strong> and <strong className="orange">API-First</strong> specs like **AsyncAPI** and **OpenAPI** to describe Inter Process Communications (IPC) for <strong className="orange">Data on the Outside</strong>.


- <strong className="orange">ZenWave Domain Language (ZDL) as Ubiquitous Language:</strong> To describe your domain core domain model.
- <strong className="orange">API-First specs like AsyncAPI and OpenAPI:</strong> to describe Inter Process Communications (IPC) between bounded contexts/microservices.
- <strong className="orange">ZenWave SDK:</strong> to generate (_a lot of_) infrastructure, functional and testing code from your models and APIs.

ZenWave SDK is designed to be easily extensible and adaptable to your project or your organization needs and likes.

## ZenWave SDK Installation

### Command Line Interface (CLI)

You can install the latest release using [jbang](https://www.jbang.dev) running the following command:

```shell
jbang alias add --force --fresh --name=zw release@zenwave360/zenwave-sdk

jbang zw --help list

```
Will output a list of all available plugins:

### Maven Plugin

You can run any available (standard or custom) plugin as part of your maven build using the maven plugin:

<details>
    <summary>Click to see the maven plugin configuration</summary>

```xml
<plugin>
    <groupId>io.zenwave360.sdk</groupId>
    <artifactId>zenwave-sdk-maven-plugin</artifactId>
    <version>${zenwave.version}</version>
    <plugin>
        <includeProjectClasspath>false</includeProjectClasspath><!-- default is false -->
        <addCompileSourceRoot>true</addCompileSourceRoot><!-- default is true -->
        <addTestCompileSourceRoot>true</addTestCompileSourceRoot><!-- default is true -->
    </plugin>
    <executions>
        <!-- Add executions for each generation here: -->
        <execution>
            <id>generate-asyncapi</id>
            <phase>generate-sources</phase>
            <goals>
                <goal>generate</goal>
            </goals>
            <plugin>
                <generatorName>spring-cloud-streams3</generatorName>
                <inputSpec>classpath:model/asyncapi.yml</inputSpec>
                <configOptions>
                    <!-- ... -->
                    <optionName>value</optionName>
                </configOptions>
            </plugin>
        </execution>
    </executions>

    <!-- add any sdk plugin (custom or standard) as dependency here -->
    <dependencies>
        <dependency>
            <groupId>io.zenwave360.sdk.plugins</groupId>
            <artifactId>asyncapi-spring-cloud-streams3</artifactId>
            <version>${zenwave.version}</version>
        </dependency>
        <dependency>
            <groupId>io.zenwave360.sdk.plugins</groupId>
            <artifactId>asyncapi-jsonschema2pojo</artifactId>
            <version>${zenwave.version}</version>
        </dependency>
    </dependencies>
</plugin>
```
Notice how you can read spec files from the project classpath as well as the filesystem. If you want to read a spec file from inside a project dependency remember to set `<includeProjectClasspath>true</includeProjectClasspath>`.

NOTE: Remember to add any plugin you want to use as dependency.
</details>

Jump to [ZenWave AsyncAPI Generator](https://www.zenwave360.io/zenwave-sdk/plugins/asyncapi-spring-cloud-streams3) for multiple examples using the maven plugin.


## ZenWave SDK Workflow

You can generate complete Event Driven Microservices using DDD and API-First principles:

> 👉 Describe your Domain Model ⤳ Generate OpenAPI ⤳ Generate AsyncAPI → Generate API Implementations → Generate Backend → Generate Tests and Contracts 👍

1. Start by <strong className="orange">Modeling your Domain</strong> using the ZDL Domain Language including: entities, relationships, service commands and domain events.
2. Generate a <strong className="orange">draft OpenAPI definition from the ZDL model</strong>. Edit collaboratively this OpenAPI document and then generate some more functional code and tests from that definition.
3. Generate a <strong className="orange">draft AsyncAPI definition</strong> for consuming async request commands and publishing domain events. Now use zenwave maven plugin to generate strongly typed business interfaces implementing some Enterprise Integration Patterns like: transactional outbox, business dead letter queue...
4. Generate a <strong className="orange">complete Backend Application</strong> from your Domain Definition Model.
5. Connect (by hand) your Backend Application to other systems using the generated OpenAPI and AsyncAPI definitions.
6. Generate <strong className="orange">E2E, Integration tests and Consumer Contracts</strong> for the public APIs you just produced.

## Generated Code Structure

Generated code follows a <strong className="orange">flexible onion/hexagonal architecture</strong>. Separating core, inbound, implementation and outbound form infrastructure and adapters.

Core domain entities and aggregates are annotated for persistence with JPA or SpringData/MongoDB annotations, <strong className="orange">avoiding unnecessary translation layers</strong> (mappers and dtos).

<Image alt="ZenWave SDK Modeling Languages" src={sdkHexagonalDark} />

Project structure:

```
📦 <basePackage>
   📦 adapters
       └─ web
       |  └─ RestControllers (spring mvc)
       └─ events
          └─ *EventListeners (spring-cloud-streams)
   📦 core
       ├─ 📦 domain
       |     └─ (entities and aggregates)
       ├─ 📦 inbound
       |     ├─ dtos/
       |     └─ ServiceInterface (inbound service interface)
       ├─ 📦 outbound
       |     ├─ mongodb
       |     |  └─ *RepositoryInterface (spring-data interface)
       |     └─ jpa
       |        └─ *RepositoryInterface (spring-data interface)
       └─ 📦 implementation
             ├─ mappers/
             └─ ServiceImplementation (inbound service implementation)
   📦 infrastructure
     ├─ mongodb
     |  └─ CustomRepositoryImpl (spring-data custom implementation)
     └─ jpa
        └─ CustomRepositoryImpl (spring-data custom implementation)
```
